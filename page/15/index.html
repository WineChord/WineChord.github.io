剑指Offer 面试题4 二维数组中的查找

发表于 2018-10-31 | 分类于 剑指Offer |

二维数组中的查找

问题描述

给一个二维数组，每一行从左到右递增，每一列从上到下递增，给一个整数，判断该整数是否在数组中。

思路

选取数组的右上角元素进行比较，如果该数字等于要查找的数字，则结束。如果大于，说明要查找元素只可能出现在该列的左侧，故剔除该列后再对剩下数组的右上角元素进行比较。如果小于，说明要查找元素只可能出现在该行的下侧，剔除该行后再对剩余数组的右上角元素进行查找。

实现

复杂度分析

每次如果没有找到，要么列减少，要么行减少，所以总的时间复杂度为 $O(m+n)$。（$m$ $n$ 为矩阵的行和列）

剑指Offer 面试题3 数组中的重复数字

发表于 2018-10-31 | 分类于 剑指Offer |

数组中的重复数字

题目一：找出数组中的重复数字

问题描述

长度为 $n$ 的数组中所有数字都在 $0$ ~ $n-1$ 范围内，找出任意重复数字。

思路

  • 可以先排序，后遍历数组，找出重复元素。时间复杂度 $O(nlogn)$
  • 还可以使用哈希表，从头到尾扫描数组，对每一个元素都可以用 $O(1)$ 时间判断哈希表中是否含有该元素。如果没有，就加入该元素，如果有，就说明找到了重复数字。其时间复杂度为 $O(n)$，需要一个 $O(n)$ 的哈希表，空间复杂度也是 $O(n)$。
  • 从头到尾扫描数组，扫描下标为 $i$ ，首先判断 $i$ 是否等于扫描的数字（记为$m$），如果相等则继续，否则判断 $m$ 和处于第 $m$ 位的数字是否相等，如果相等则表明出现了重复数字，不相等则交换 $m$ 和 $i$ ，重复对当前数的判断。

    实现

复杂度分析

尽管有两重循环，但是每个数字最多只要进行两次交换即可到位，因此总的时间复杂度为 $O(n)$。操作未用额外的空间，故空间复杂度为 $O(1)$。

题目二：不修改数组找出重复的数字

问题描述

数组长度为 $n+1$，所有数字都在 $1$ ~ $n$ 范围内，要求不能修改原数组找出重复元素。

思路

  • 可以开辟一个辅助数组，将原数组的元素 $m$ 复制到新数组第 $m$ 位处。空间复杂度为 $O(n)$。
  • 为了降低空间复杂度，考虑使用二分法，$1$ ~ $n$ 分为 $1$ ~ $m$ 和 $m+1$ ~ $n$ 两部分，如果整个数组元素在 $1$ ~ $m$ 范围中的计数大于 $m$，说明该范围内含有重复数字，对于后半部分同理。然后再对前半部分或后半部分进行二分处理。

    实现

复杂度分析

$countRange$ 函数要调用 $O(logn)$ 次，每次需要 $O(n)$ 时间，总的时间复杂度是 $O(nlong(n))$，空间复杂度为 $O(1)$。与之前的方法相比属于用时间换空间。

但是注意这个算法不能保证找到所有重复数字。比如他不能找出 { $2,3,5,4,3,2,6,7$ }中的重复数字 $2$（$3$ 是可以找到的）。这是因为在 $1$ ~ $2$ 中刚好有两个数字，所以无法确定是每个数字出现了一次还是其中的一个数字出现了两次。

6.004 Fall 2015 L11: Compilers

Implementations: Interpretation vs compilation

Interpretation

Model of Interpretation:

  • Start with some hard-to-program machine, say $M_1$
  • Write a single program for $M_1$ that mimics the behavior of some easier machine, $M_2$
  • Result: a “virtual” $M_2$

Layers of interpretation:

  • Often we use several layers of interpretation to achieve desired behavior.
  • x86 CPU, running
    • Python, running
      • Application, interpreting
        • Data.

Compilation

Model of Compilation

  • Given some hard-to-program machine, say $M_1$…
  • Find some easier-to-program language $L_2$ (perhaps for a more complicated machine, $M_2$); write programs in that language
  • Build a translator (compiler) that translates programs from $M_2$$’s$ language to $M_1$$’s$ language. May run on $M_1$, $M_2$, or some other machine.

Interpretation and compilation: two ways to execute high-level languages

  • Both allow changes in the programming labguage
  • Both afford programming applications in platform (e.g., processor) independent languages
  • Both are widely used in modern computer systems!

Interpretation vs Compilation

  • Characteristic differences:
\ Interpretation Compilation
How it treats input “x+2” Computes x+2 Generates a program that computes x+2
When it happens During execution Before execution
What it complicates/slows Program execution Program development
Decisions made at Run time Compile time
  • Major choice we’ll see repeatedly: do it at compile time or at un time?
    • Which is faster?
    • Which is more general?

Compilers

  • Bare minimum for a functional compiler:
  • Good compilers:
    • Produce meaningful errors on incorrect programs
      • Even better: meaningful warnings
    • Produce fast, optimized code
  • This lecture:
    • Simple techniques to compile a C procedure into assembly
    • Overview of how modern compilers work

Anatomy of a Modern Compiler
Read source program Optimize IR
Break it up into basic elements Translate IR to ASM
Check correctness, produce errors Optimize ASM
Translate to generic intermediate representation(IR)

Frontend Stages

  • Lexical analysis (scanning): Source -> List of tokens
  • Syntactic analysis (parsing): Tokens -> Syntax tree
  • Semantic analysis (mainly, type checking)

Intermediate Representation (IR)

  • Internal compiler language that is:
    • Language-independent
    • Machine-independent
    • Easy to optimize
  • Why yet another language?
    • Assembly does not have enough info to optimize it well
    • Enables modularity and reuse

Common IR: Control Flow Graph

  • Assignments:

  • Basic block: Sequence of assignments with an optional branch at the end\

  • Control flow graph:

    • Nodes: Basic blocks
    • Edges: Jumps or branches between basic blocks

Control Flow Graph for GCD

IR Optimization

  • Perform a set of passes over the CFG
    • Each pass does a specific, simple task over the CFG
    • By repeating multiple simple passes on the CFG over and over, compilers achieve very complex optimizations
  • Example optimizations:
    • Dead code elimination: Eliminate assignments to variables that are never used, or basic blocks that are never reached
    • Constant propagation: Identify variables that are constant, substitute the constant elsewhere
    • Constant folding: Compute and substitute constant expressions

Example IR Optimizations


  • More optimizations by adding passes: Common subexpression elimination, loop-invariant code motion, loop unrolling…

Code Generation

  • Translate generated IR to assembly
  • Register allocation: Map variables to registers
    • If variables > registers, map some to memory, and load/store them when needed
  • Translate each assignment to instructions
    • Some assignments may require > 1 instr if our ISA doesn’t have op
  • Emit each basic block: label, assignments, and branch or jump
  • Lay out basic blocks, removing superfluous jumps
  • ISA and CPU-specific optimizations
    • e.g., I possible, reorder instructions to improve performance

Description

If another lighthouse is in grey area, they can beacon each other.

Input

  • First line: the total number of lighthouse N
  • Second to (N+1)th line: lighthouse is on the point (X, Y)

Output

How many pairs of lighthouses can beacon each other.

Restrictions

  • 90% : 1 <= n <= 3 * 10^5
  • 95% : 1 <= n <= 10^6
  • All : 1 <= n <= 4 * 10^6
  • For every lighthouses, X coordinates won’t be the same, Y coordinates won’t be the same.
  • 1 <= x, y <= 10^8
  • Time: 2s
  • Memory: 256 MB

Example

  • input

  • output

Idea

It is trivial to think that use two for loops can solve the problem. However the time consumption will be too large. First construct a point struct. Use quick sort to make the point is sorted according to it X coordinate. Then calculate the inversions in the Y coordinates. The total inversions mean that they cannot form lighthouses that beacon each other. Thus the result equals to the total permutation number minus the total inversions.
The hard point is how to calculate inversions. Here uses a variation of merge sort. When doing mergesort to their Y coordinates, keep a count value to record inversions.
