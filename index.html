发表于 2019-02-20 | 分类于 CS 276 IR |

主要讲索引的结构

  • Index construction
    • how?
    • limited main memory?

Hardware basics

  • access in memory is much faster than on disk

  • Disk I/O is block-based: 8 KB to 256 KB

  • Hardware assumptions

(s) ave. seek time: 5ms

(b) transfer time per byte: 0.02 us

processor's clock rate: 10^9 Hz

(p) low-level operation: 0.01 us

size of main memory: several GB

size of disk: 1 TB or more

  • Our collection for this lecture: RCV1
    • One year of Reuters newswire

(N) doc: 800,000

(L) avg. $ tokens per doc: 200

(M) terms(word types): 400,000

avg. # bytes per token(includ. spaces/punct.): 6

avg. # bytes per token(without s/p): 4.5

avg. # bytes per term: 7.5

non-positional postings: 100,000,000

  • Recall IIR 1 index construction

  • Key step: inverted file is sorted by terms

    • focus on this sort step: 100M items to sort

  • Scaling index construction

    • In-memory index construction does not scale
    • How to construct an index for very large collections?

  • Sort-based index construction

    • As we build the index, we parse docs one at a time
    • The final postings for any term are incomplete until the end
    • At 12 bytes per non-positional postings entry(term, doc, freq), demands a lot of space for large collections.
    • T = 100,000,000 in the case of RCV1
    • Thus: We need to store intermediate results on disk

  • Sort using disk as "memory"?

    • Too slow: many disk seeks
    • need external sorting algorithm

  • Bottleneck

    • Parse and build postings entries one doc at a time
    • Now sort postings entries by term (then by doc within each term)
    • Doing this with random disk seeks would be too slow

  • BSBI: Blocked Sorted-Based Indexing(Sorting with fewer disk seeks)

    • 12-byte (4+4+4) records (term, doc, freq)
    • These are generated as we parse docs
    • Must now sort 100M such 12-byte records by term
    • Define a Block ~10M such records
      • 10 such blocks
    • Basic idea of algorithm:
      • Accumulate postings for each block, sort, write to disk
      • Then merge the blocks into one long sorted order

  • Sorting 10 blocks of 10M records

    • First, read each block and sort within:
      • Qsort: 2NlnN expected steps
    • 10 times this estimate -- gives us 10 sorted runs of 10M records each
    • Done straightforwardly, need 2 copies of data on disk
      • can be optimized

  • How to merge the sorted runs?

    • Can do binary merges
    • more efficient to do a multi-way merge

  • Remaining problem with sort-based algorithm

    • assumption: can keep the dictionary in memory
    • dictionary to implement a term to term ID mapping
    • but: could work with term instead of termID
      • intermediate files become very large

  • SPIMI: Single-Pass In-Memory Indexing

    • Key idea1: Generate separate dictionaries for each block -- no need to maintain term-termID mapping across blocks
    • Key idea2: Don't sort. Accumulate postings in postings lists as they occur
    • With these two ideas we can generate a complete inverted index for each block
    • These separate indexes can then be merged into one big index

  • SPIMI: Compression

    • makes it even more efficient
      • Compression of terms
      • Compression of postings

  • Distributed indexing

    • web-scale indexing: use a distributed computing cluster

  • Web search engine data centers

  • Massive data centers

  • Distributed indexing

    • Maintain a master machine directing the indexing job
      • -- considered "safe"
    • Break up indexing into sets of (parallel) tasks
    • Master machine assigns each task to an idle machine from a pool

  • Parallel tasks

    • use two sets of parallel tasks
      • Parsers/Inverters
    • Break the input document collection into splits
    • Each split is a subset of documents(corresponding to blocks in BSBI/SPIMI)

  • Parsers

    • Master assigns a split to an idle parser machine
    • Parser reads a document at a time and emits (term, doc) pairs
    • Parser writes pairs into j partitions
    • Each partition is for a range of terms' first letters
      • a-f, g-p, q-z ( j=3 )
    • Now to complete the index inversion

  • Inverters

    • An inverter collects all (term, doc) pairs (= postings) for one term-partition
    • Sorts and writes to postings lists

  • MapReduce

    • Google indexing system consists of a number of phases, each implemented in MapReduce.
    • Index construction was just one phase
    • Another phase: transforming a term-partitioned index into a document-partitioned index
      • Term-partitioned: one machine handles a subrange of terms
      • Document-partitioned: one machine handles a subrange of documents
    • in the web part of the course, most search engines use a document-partitioned index

  • Schema for index construction in MapReduce

    • Schema of map and reduce functions
    • map: input $\rightarrow$ list(k,v), reduce: (k, list(v)) $\rightarrow$ output
    • Instantiation of the schema for index construction
    • map: collection $\rightarrow$ list(termID, docID)
    • reduce: (<termID1, list(docID)>, <termID2, list(docID)>,...) $\rightarrow$ (postings list1, postings list2,...)

  • Dynamic indexing

    • all above... are static
    • Doc being added or deleted or modified

  • Simplest approach

    • Maintain "big" main index
    • New docs go into "small" auxiliary index
    • Search across both, merge results
    • Deletions
      • Invalidation bit-vector for deleted docs
      • Filter docs output on a search result by this invalidation bit-vector
    • Periodically, re-index into one main index

  • Issues with main and auxiliary indexes

    • Problem of frequent merges -- you touch stuff a lot
    • Poor performance during merge
    • Assumption for the rest of the lecture: The index is one big file
    • In reality: Use a scheme somewhere in between(e.g., split very large postings lists, collect postings lists of length 1 in one file etc.)

  • Logarithmic merge

    • Maintain a series of indexes, each twice as large as the previous one
    • Keep smallest ($Z_0$) in memory
    • Larger ones ($I_0,I_1,...$) on disk
    • If $Z_0$ gets too big ($\gt n$), write to disk as $I_0$
    • or merge with $I_0$ (if $I_0$ already exists) as $Z_1$
    • Either write merge $Z_1$ to disk as $I_1$ (if no $I_1$)
    • Or merge with $I_1$ to form $Z_2$

  • Further issues with multiple indexes

    • Collection-wide statistics are hard to maintain

  • Dynamic indexing at search engines

    • All the large search engines now do dynamic indexing
    • Their indices have frequent incremental changes
    • But they also periodically reconstruct the index from scratch

  • Other sorts of indexes

    • Positional indexes
      • Same sort of sorting problem ... just larger
    • Building character n-gram indexes:
      • As text is parsed, enumerate n-grams
      • For each n-gram, need pointers to all dictionary terms containing it -- the "postings"
      • Note that the same "postings entry" will arise repeatedly in parsing the docs -- need efficient hashing to keep track of this

Introducing Information Retrieval and Web Search

  • Def.
    • IR is finding material(usually documents) of an unstructured nature(usually text) that satisfies an information need from within large collections(usually stored on computers).
    • Collection: A set of documents
    • Goal: Retrieve documents with info. that is relevant to the user's info. need and helps the user complete a task.
  • Eval: How good are the retrieved docs?
    • Precision: Fraction of retrieved docs that are relevant to the user's info. need
    • Recall: Fraction of relevant docs in collection that are retrieved.

Term-document incidence matrices

  • using the number of appearances in each document to represent a term.

The Inverted Index

  • For each term t, store a list of all documents that contains t.

    • Identify each doc by a docID

  • Need variable-size postings lists (inside it is sorted by docID)

    • On disk: a continuous run of postings.
    • In memory: linked lists or variable length arrays

  • Inverted index construction

    • Documents to be indexed
    • $\underrightarrow{Tokenizer}$
    • Token stream
    • $\underrightarrow{Linguistic modules}$
    • Modified tokens
    • $\underrightarrow{Indexer}$
    • Inverted index

  • Initial stages of text processing

    • Tokenization
    • Normalization
    • Stemming
    • Stop words

  • Indexer steps

    • Token sequence
    • Sort (by terms and then docID)
    • Dictionary & Postings (Doc. frequency info is added)

  • Problems taken into consideration

    • efficient index
    • storage

Query processing with an inverted index

  • Query processing: AND
    • "Merge" the two postings: O(x+y)
    • Crucial: postings sorted by docID

The Boolean Retrieval Model & Extended Boolean Models

  • Boolean queries: Exact match
    • queries using AND, OR and NOT to join query terms
      • View each doc as a set of words
      • Is precise
  • Query optimization
    • Process in order of increasing freq
      • start with smallest set, then keep cutting further

Phrase queries and positional indexes

  • to answer queries such as "Stanford university"
  • It is no longer suffices to store only <term: docs> entries
  • A first attempt: Biword indexes
    • Index every consecutive pair of terms in the text as a phrase
    • Problems:
      • False positives
      • Index blowup due to bigger dictionary
  • Solution 2: Positional indexes

<term, number of docs containing term;

doc1: position1, position2...;

doc2: position1, position2...;

etc.>

  • Processing a phrase query
    • Extract inverted index entries for each distinct term
    • Merge their doc
  • Proximity queries
  • Positional index size
    • Although expands postings storage substantially, it is now standardly used because of the power and usefulness of phrase and proximity queries.
  • Rules of thumb
    • 2-4 as large as a non-positional index
    • size 35-50% of volume of original text
  • Combination schemes
    • can be profitably combined

Structured vs. Unstructured Data

  • Structured: info. in tables
  • Unstructured: free text
  • Semi-structured: has distinctly identified zones

最佳图搜 $A^*$ （Optimal Search）

  • 当 $h(n)$ 是处在 $h*(n)$ 的下界范围，即 $h(n)\le h*(n)$ 时，称之为 $A^*$
  • 实际上是分支界限和DP以及使用下界范围的 $h$ 相结合的算法
  • $h(n)\equiv 0$ 时，$g\equiv d$，为宽度优先
  • 当 $s$ 到目标结点有一条路径存在时，若搜索算法总是在找到一条从 $s$ 到目标结点的最佳路径上结束，则称该搜索算法是可采纳的（Admissibility）
  • $A^*$ 即有可采纳性
  • 定理1：对有限图，若 $s\rightarrow t$ 存在路径，则 $A$ 定成功
  • 引理2.1：对无限图，若有 $s\rightarrow t$ 的路径，则 $A^$ 不结束时，在 OPEN 中即使最小的一个 $f$ 值也将增加到任意大，或有 $f(n)\gt f(s)$
  • 引理2.2：$A^$ 结束前，OPEN 中必存在 $f(n)\le f(s)$ 的结点，$n$ 在最佳路径上
  • 定理2：对无限图，若 $s\rightarrow t$ 存在路径，则 $A^*$ 也一定成功结束
  • 推论2.1：OPEN 中任一 $f(n)\lt f*(s)$ 的结点 $n$ 都会被 $A^*$ 选为扩展
  • 定理3：若 $s\rightarrow t$ 存在路径，则 $A^*$ 必能找到最优解
  • 推论3.1：$A^$ 选择扩展的结点 $n$，有 $f(n)\le f(s)$
  • 定理4：两个 $A^*$ 算法：$A1,A2$ ，若 $A2$ 比 $A1$ 有较多启发信息（$h2(n)\gt h1(n)$），则搜索结束时，$A1$ 扩展的结点至少和 $A2$ 一样多

$A^*$ 改进

  • $A$ 算法中可能会出现重复多次扩展同一个节点的现象

  • 若对启发函数施加一定的限制——单调限制，则当 $A^*$ 算法选某一个结点扩展时，就已经找到该结点的最佳路径

  • 一个启发函数 $h$，如果对所有节点 $ni$ 和 $nj$ （$nj$ 是 $ni$ 的子节点），都有 $h(ni)-h(nj)\le C(ni,nj)$ 且 $h(ti)=0$，则称 $h$ 满足单调限制条件

  • 即：从 $ni$ 到目标结点，最佳路径耗散值估计 $h(ni)$ 不大于 $nj$ 到目标结点最佳路径耗散值估计 $h(nj)$ 与 $ni$ 到 $nj$ 弧耗散值两者之和

  • 易证：满足单调条件的 $h$，一定满足 $A^*$ 条件，但反过来不一定成立

  • 对于八数码问题，$h(n)=W(n)$ 满足单调限制条件

  • 证明：

  • 要证明 $h$ 单调，只需证 $h(ni)-h(nj)\le C(ni,nj)$，且 $h(ti)=0$，当用“不在位的将牌数”来定义 $h$ 时，显然 $h(ti)=0$；而对于每一次的移动，会出现以下三种情况：

    1. “在位”$\rightarrow$ “不在位”，此时 $h(ni)-h(nj)=-1$
    2. “不在位”$\rightarrow$ “不在位”，此时 $h(ni)-h(nj)=0$
    3. “不在位”$\rightarrow$ “在位”，此时 $h(ni)-h(nj)=1$

  • 综上均有：$h(ni)-h(nj)\le 1$，而 $C(ni,nj)=1$，故第一条也满足。因此该 $h$ 单调

  • 定理5：若 $h(n)$ 满足单调限制条件，则 $A^$ 扩展了结点 $n$ 之后，就已经找到了到达结点 $n$ 的最佳路径，即若 $A^$ 选 $n$ 来扩展，在单调限制条件下有 $g(n)=g*(n)$

  • 定理6：若 $h(n)$ 满足单调限制，则由 $A^*$ 所扩展的结点序列，其 $f$ 值是非递减的，即 $f(ni)\le f(nj)$

  • 改进：
  • 按照结点的 $f$ 值从小到大排序，以 $f*(s)$ 为界将 OPEN 划分为两个部分，$f\lt f*(s)$ 的称为 $NEST$
  • 由推论2.1，OPEN 中 $f(n)\lt f*(s)$ 的结点 $n$ 一定会被扩展，故 $NEST$ 中的结点一定会被扩展，对于这些点，可以设他们的 $h=0$，即按照 $f(n)=g(n)$ 来进行扩展。
  • $f*(s)$ 的值如何确定：用目前为止扩展的结点中最大的 $f$ 来作为 $f*(s)$ 的近似值，记为 $fm$，这个值动态变化，只有在 $NEST$ 为空时才进行更新（因为 $NEXT$ 中的 $f$ 均小于 $fm$）

  • 产生系统的三要素：综合数据库（Global Database），产生规则集（Set of Rules），控制系统（Control System）
  • 综合数据库：一个数据的集合，用于存放在推理过程中的已知条件，推导出的中间结果和最终结论等
  • 产生规则集：相当于知识库，IF 前件 THEN 后件，来表达求解问题需要的知识
    • 规则可以表达求解问题时的客观规律，如“马走日，象走田”
    • 也可以是对求解问题有帮助的策略方面的知识，如“什么局面有利”
  • 控制系统：又称控制策略或搜索策略，用于控制系统的运行，根据综合数据库当前数据来选择合适的规则，也称推理引擎
  • （Production System）主要是使用类似于文法的规则，对符号串作替换运算
  • 产生式系统模拟了人类的思考过程
  • 特点：
    • 数据驱动：系统在数据的变化下进行行为的变化
    • 独立性：三要素之间是相互独立的

一阶谓词演算的基本体系

  • 谓词演算是一种形式语言，是AI中一种常用的知识表示方法，可以表示各种描述语句。如在产生式系统中，用来表达综合数据库，规则集的描述
  • 谓词演算体系中的一些演绎推理方法，还可以用来建立自动定理证明系统，问答系统，基于规则的演绎系统等
  • 含：标点符号，括号，逻辑联结词，常量符号集，变量符号集，$n$ 元函数符号集，$n$ 元谓词符号集，量词（全称$\forall$ 存在$\exists $）
  • 并定义了谓词演算的合法表达式（原子公式，合式公式wff），表达式的演算化简方法，以便将一般化的表达式化成标准式（合取的前束范式或析取得前束范式）
  • 化简结果的标准式：$F\triangleq(Q_1x_1)...(Q_nx_n)M$
    • 其中 $(Q_1x_1)...(Q_nx_n)$ 为前束，代表各种量词的约束关系
    • $M$ 称为母式，是不包含量词符号量化的合式公式范式
  • 谓词演算中建立有很多推理规则可用来从已知的公式集推出新的公式，这些导出的公式称为定理。
  • 给出定理的推理过程及所使用的推理规则序列就构成了该定理的一个证明
  • 在证明定理的演绎过程中，经常要对量化的表达式进行匹配操作，因而需要对项作变量置换使表达式一致起来，该过程称作合一
  • 前束范式：所有量词均非否定地出现在公式的前部，且所有的量词的约束范围均是整个公式
  • 任何一个合式公式均可以转化为一个前束范式
  • 消去前束范式中所有的存在量词后得到的合式公式称为 $S$ 范式，这一过程称为 $skolem$ 化
  • $S$ 范式与原式不一定等价，但在不可满足方面是等价的
  • 利用结合律，分配律等可以将 $S​$ 范式转化为一个合取范式，合取范式是前束范式的一种，其母式具有如下的形式：$A_1\and A_2\and ...\and A_n​$
  • 其中 $A_i​$ 是由原子公式或原子公式的否定组成的析取项。每个析取项又称之为子句，一个合取范式可以用如下的子句的集合（子句集）表示：

${A_1,A_2,...A_n}​$

  • 例：

$(x)(y)[(P(x,y)\or \sim Q(x,y))\and (\sim P(x,y)\or U(x)\or V(y))\and(Q(x,y)\or \sim U(y))]​$

即为一个合取范式，其子句集为：

${P(x,y)\or \sim Q(x,y),\sim P(x,y)\or U(x)\or V(y), Q(x,y)\or \sim U(y)}$

  • 任何一个合式公式均可以通过九步来化为子句集，例：

$(z)(x)(y){[(P(x)\or Q(x))\rightarrow R(y)]\or U(z)}​$

$A(x)\rightarrow B(x) \equiv \sim A(x)\or B(x)​$

有：

$(P(x)\or Q(x))\rightarrow R(y)\Rightarrow \sim (P(x)\or Q(x))\or R(y)$

代入式中有：

$(\exists z)(\forall x)(\exists y){[\sim(P(x)\or Q(x))\or R(y)]\or U(z)}$

  1. 移动否定符

应用摩根定律：

$\sim(A(x)\or B(x))\equiv \sim A(x)\and \sim B(x)$

$\sim(A(x)\and B(x))\equiv \sim A(x)\or \sim B(x)$

故而有：

$\sim(P(x)\or Q(x))\Rightarrow \sim P(x)\and \sim Q(x)$

代入：

$(\exists z)(\forall x)(\exists y){[(\sim P(x)\and\sim Q(x))\or R(y)]\or U(z)}$

若否定作用于量词，则根据量词转换率将量词提到否定的作用域外：

$\sim((\forall x)A(x))\equiv (\exists x)(\sim A(x))$

$\sim ((\exists x)A(x))\equiv (\forall x)(\sim A(x))$

本例未出现此情况。

  1. 变量换名

在不同的量词约束的变量，应使用不同的变量名

$(\forall x)A(x)\or (\exists x)B(x)$

改为：$(\forall x)A(x)\or (\exists y)B(y)$

  1. 量词左移

将所有量词移到最左侧，但是不改变次序，如上例可以重写为：

$(\forall x)(\exists y){A(x)\or B(y)}$

这也是要在第三步改名的原因，否则就不能移动

至此，一个合式公式便转化为了前束范式，特点为：所有量词均非否定地出现在公式的前部，并且所有量词的约束范围均是整个合式公式。

  1. 消去存在量词（skolem化）

消去存在量词的规则如下：

  • 假设 $\Psi$ 为前束范式中的一个存在量词，若其前面没有全称量词，则将 $\Psi$ 所约束的变量 $x$ 用一个全新的常量代替
  • 若前面有全称量词，则将 $x$ 用一个新的函数（称为 $skolem$ 函数）代替，该函数的变量是那些在 $\Psi$ 前面的全称量词所约束的变量，然后将存在量词 $\Psi$ 消去

$\Rightarrow (\forall x){[(\sim P(x)\and \sim Q(x))\or R(f(x))]\or U(a)}$

利用结合律，分配律等将 $S$ 范式的母式转化为合取范式

$(A(x)\and B(x))\and C(x)\equiv A(x)\and(B(x)\and C(x))$

$(A(x)\or B(x))\or C(x)\equiv A(x)\or (B(x)\or C(x))$

$A(x)\and (B(x)\or C(x))\equiv (A(x)\and B(x))\or(A(x)\and C(x))$

$A(x)\or(B(x)\and C(x))\equiv (A(x)\or B(x))\and(A(x)\or C(x))$

对于本例：

$(\forall x){[(\sim P(x)\and\sim Q(x))\or R(f(x))]\or U(a)}$

$\Rightarrow (\forall x){[\sim P(x)\and\sim Q(x)]\or R(f(x))\or U(a)}$

$\Rightarrow (\forall x){[\sim P(x)\or R(f(x))\or U(a)]\and[\sim Q(x)\or R(f(x))\or U(a)]}$

  1. 隐去全称量词

此时所有的变量都是受全称量词约束，故而可以将全称量词隐去，默认所有的变量都是受全称量词约束的。

$[\sim P(x)\or R(f(x))\or U(a)]\and [\sim Q(x)\or R(f(x))\or U(a)]$

  1. 表示为子句集

用 "," 代替公式中的 "$\and$"，并用 "{}" 括起来

${\sim P(x)\or R(f(x))\or U(a),\sim Q(x)\or R(f(x))\or U(a)}$

  1. 变量换名

使不同子句中的变量使用不同的名字

${\sim P(x_1)\or R(f(x_1))\or U(a),\sim Q(x_2)\or R(f(x_2))\or U(a)}$

  2. 消去蕴涵符
  3. 将否定词只作用于原子公式为止
  4. 变量标准化（进行必要换名）
  5. 所有量词左移，不要改次序
  6. 消去存在量词

$F=(\exists x)(\exists y)(\forall z)(\exists u)(\forall v)(\exists w)(P(x,y,z,u,v,w)$

$\and (Q(x,y,z,u,v,w)\or\sim R(x,z,w)))$

四个存在量词替换如下：

$x=a,y=b,u=f(z),w=g(z,w)$

得：

$F1=(\forall z)(\forall v)(P(a,b,z,f(z),v,g(z,v))$

$\and (Q(a,b,z,f(z),v,g(z,v))\or\sim R(a,z,g(z,v))))$

  1. 将母式化为合取范式。反复使用结合律和分配律，将母式表达成合取范式的标准型，最后得到一个 $Skolem$ 化的前束范式 $Fs$
  2. 省略全称量词
  3. 将母式用子句集表示

$$ S={ P(a,b,z,f(z),v,g(z,v)),Q(a,b,z,f(z),v,g(z,v))\or\sim R(a,z,g(z,v))} $$

  1. 子句变量标准化，使任意两个子句不会有相同的变量出现。
  • 标准式应用问题
  • 经过上述步骤化简之后得到的标准式是经过 $Skolem$ 化的前束范式，通常也称为 $S-标准形$ ，注意其不唯一，记为 $Fs$
  • 定理：若 $S$ 是合取公式 $F$ 的 $S-$标准形之子句集，则 $F$ 为永假的充要条件是 $S$ 为不可满足的。
  • 子句：为标准形母式中的合取元，因而每一个子句是若干文字的析取式，不含有文字的子句称为空子句
  • 空子句取值总为假，记为 NIL
  • 在一个子句的文字中，用常量代替变量，得到一个文字的基例，由基文字组成的子句称为基子句
  • 子句集中所有元素（即子句）的合取式不为真，则称该子句集为不可满足的

归结

  • 一个重要的推理规则
  • 1965年 Robinson 提出，使自动定理证明技术得到很大发展
  • 先介绍归结方法的基本思想
  • 再分别讨论命题逻辑和谓词逻辑的归结原理

归结原理

  • 类似反证法
  • 先对结论求反，然后将已知条件和结论的否定合在一起用子句集表达
  • 若该子句集存在矛盾，则证明了结论的正确性
  • 证明子句集存在矛盾的方法：
    • 设 $S$ 是已知条件和结论的否定合并后对应的子句集
    • 假设有一种变换方法，可以对 $S$ 实施一系列的变换，而且该变换能够保证变换前后的子句集在不可满足的意义下是等价的
    • 如此，若最终得到的子句集是不可满足的，就证明了子句集 $S$ 是不可满足的，从而结论成立
  • 子句集中的子句是“与”的关系，如果在子句集中出现了空子句，则该子句集不可满足。
  • 故归结过程就是寻找空子句的过程
  • 若看为搜索，则初始状态为已知条件和结论的否定对应的子句集，目标为空子句，规则是归结
  • 证明 $F=F1\and F2\and...\and Fn\rightarrow W$ 为永真
  • 等价于证明 $F=F1\and F2\and...\and Fn\and \sim W$ 为永假
  • 等价于证明 $F$ 对应的子句集 $S=S0 \cup {\sim W}$ 为不可满足的
  • 此时若用归结作为推理规则使用时，可以简化机器证明
  • 思路：设法检验扩充的子句集 $Si$ 是否含有空子句，若 $S$ 集中存在空子句，则表明 $S$ 为不可满足的。若无空子句，则进一步用归结法从 $S$ 导出 $S1$ 然后再检验 $S1$ 是否有空子句，可以证明用归结法导出的扩大子句集 $S1$，其不可满足性是不变的，所以若 $S1$ 中有空子句，也就证得了 $S$ 不可满足性。
  • 归结一直进行下去，目的是通过归结过程演绎出 $S$ 额不可满足性，从而使定理得到证明

命题逻辑的归结原理

  • 命题逻辑：不含变量的逻辑
归结式（或预解式）的定义及性质
  • 归结式：对于任意两个子句 $C1$ 和 $C2$ ，若 $C1$ 中有一个文字 $L1$，而 $C2$ 中有一个与 $L1$ 成互补的文字 $L2$ ，分别从 $C1,C2$ 中删除 $L1$ 和 $L2$，并将其剩余部分组成新的析取式，则称新子句为归结式或预解式
  • 例：$C1=L\or C1'$ $C2=(\sim L)\or C2'$，则归结式 $C=C1'\or C2'$，当 $C1'=C2'=NIL$ 时，$C=NIL$
  • 定理：两个子句 $C1$ 和 $C2$ 的归结式 $C$ 是 $C1$ 和 $C2$ 的逻辑推论
  • 证：略
  • 推论：子句集 $S={C1,C2,...,Cn}$与子句集 $S1={C,C1,C2,...,Cn}$的不可满足性是等价的（$C$ 为 $C1$ 和 $C2$ 的归结式）
  • 证：
    • 设 $S$ 是不可满足的，则 $C1,C2,...,Cn$ 中必有一为假，因而 $S1$ 必为不可满足的
    • 设 $S1$ 是不可满足，分两种情况：
      • $C$ 为真：则 $C1,C2,...,Cn$ 中必有一为假，故 $S$ 不可满足
      • $C$ 为假： 则 $C1\or C2$ 为假，即 $C1$ 为假或 $C2$ 为假，故 $S$ 也是不可满足的
    • 因此 $S$ 和 $S1$ 的不可满足性是等价的
  • 同理可证 $Si$ 和 $Si+1$ 的不可满足性也是等价的（$i=1,2,...$）
  • 归结原理即为从子句集 $S$ 出发，应用归结推理规则导出子句集 $S1$ ，再从 $S1$ 出发导出 $S2$ ，直到某一个子句集 $Sn$ 出现空子句为止。
  • 根据不可满足性等价原理，已知若 $Sn$ 为不可满足的，则可逆向依次推得 $S$ 必为不可满足的
  • 由此可见，用归结法证明定理，过程比较单纯，只涉及归结推理规则的应用问题，因而便于实现机器证明
命题逻辑的归结过程
  • 给定公理集 $F$ 命题集 $P$，归结过程如下：
  1. 把 $F$ 转化为子句集表示，得子句集 $S0$
  2. 把命题 $P$ 的否定式 $\sim P$ 也转化为子句集表示，并将其加到 $S0$ 中，得 $S=S0\cup {S\sim p}$
  3. 对子句集 $S$ 反复应用归结推理规则，直至导出含有空子句的扩大子句集为止，即出现归结式为空子句的情况，表明已经找到了矛盾，证明过程结束

例：

已知公理集为：

$P$ (1)

$(P\and Q)\rightarrow R$ (2)

$(S\or T)\rightarrow Q$ (3)

$T$ (4)

求证 $R$

  • 由 (1) 有子句集 ${P}$
  • 由 (2) 有：
    • $(P\and Q)\rightarrow R$
    • $\Rightarrow \sim(P\and Q)\or R$
    • $\Rightarrow (\sim P\or \sim Q\or R)$
    • 得子句集：${\sim P\or \sim Q\or R}$
  • 由 (3)
    • $(S\or V)\rightarrow Q$
    • $\Rightarrow \sim (S\or T)\or Q$
    • $\Rightarrow (\sim S\and \sim T)\or Q$
    • $\Rightarrow (\sim S\or Q)\and (\sim T\or Q)$
    • 得子句集：${\sim S\or Q,\sim T\or Q}$
  • 由 (4) 有子句集${T}$
  • 由结论的否定得子句集：${\sim R}$
  • 将以上子句集并在一起有总的子句集：
    • ${P,\sim P\or\sim Q\or R,\sim S\or Q,\sim T\or Q,T,\sim R}$

谓词逻辑的归结原理

  • 命题逻辑中较容易确定用于归结的两个归结对
  • 而对于谓词逻辑来说，由于谓词具有常量，变量和函数等变元的存在，两个子句是否可用于归结，判断起来是比较复杂的
  • 如：
    • $P(x)\or Q(y)$ 和 $\sim P(f(y))\or R(y)$ 是否可以归结？
    • $P(A)\or Q(y)$ 和 $\sim P(f(y))\or R(y)$ 是否可以归结？
    • 后面可看出，前者可以归结而后者不可以归结
  • 从命题逻辑的归结过程可以看出，求归结式时比较容易确定含有互补对文字的两个子句，因而匹配过程比较简单
  • 但是在谓词逻辑中，需要考虑变量的约束问题，所以往往要对公式作变量置换和合一等处理，才能得到互补的基本式
  • 过程为：
    • 若 $S$ 中两子句间有相同互补文字的谓词，但他们的项不同，则必须找出对应的不一致项
    • 进行变量置换，使其对应项一致
    • 求归结式看能否导出空子句
合一（Unify）
  • 一个有序对集合 $s={ti/vi}$ $(i=1,2,...,n)$ 称为置换
  • 其中 $vi$ $(i=1,2,...,)$ 是变量， $ti$ $(i=1,2,...,n)$ 是不同于 $vi$ 的项，可以是常量，函数或其他变量
  • 置换 $s$ 表示将公式中所有变量 $vi$ 用 $ti$ 代替
  • 对公式 $E$ 施以置换 $s$，记作 $Es$，$Es$ 称为公式 $E$ 的例

例：设有置换 $s={z/x,A/y}$，

则：$P[x,f(y),B]s=P(z,f(A),B)$

设有公式的集合 ${Ei}$ $(i=1,2,...,m)$，如果存在一个置换 $s$，使得这 $m$ 个公式被施以 $s$ 以后，变得完全一样了，则称这 $m$ 个公式是可合一的，置换 $s$ 是它们的合一者

例：

设有公式集 ${P(x,f(y),B),P(z,f(B),B)}$ 和置换 $s1={A/x,B/y,A/z}$

由于

$P(x,f(y),B)s=P(A,f(B),B)$

$P(z,f(B),B)s=P(A,f(B),B)$

所以公式集是可合一的，置换 $s1$ 是它们的合一者

一般来说合一者不唯一

  • 对于一个公式集来说，如果存在几个合一者，则其中置换数最少，限制最少，产生的例最具一般性的置换称为一般合一者（mgu）
  • 一般合一者也不一定唯一

$s1=(z/x,w/y)$ $P[x,f(y),B]s1=P[z,f(w),B]$

$s4=(c/x,A/y)$ $P[x,f(y),B]s4=P[C,f(A),B]$

  • 第一个例为原始文字初等变式，实际上置换后只是对变量做了换名
  • 第四个例为基例，置换后不再含有变量
  • 通常用有序对的集合 $s={t_1/v_1,t_2,v_2,...,t_n/v_n}$来表示任一置换，用 $s$ 对表达式 $E$ 作置换后的例简记为 $Es$。
  • 有时要进行多次替换，可以进行合成

如：

$s1={g(x,y)/z}$，$s2={A/x,B/y,C/w,D/z}$

$s1s2={g(A,B)/z,A/x,B/y,C/w}$

  • UNIFY 算法用来判断两个公式是否可以合一，并且在可合一的情况下，给出他们的 mgu
  • 基本思想（用递归描述）：
    • 从左向右按位置扫描两个谓词的每一个对应部分
    • 如果二者一致，则转向下一项
    • 若不一致，查看是否存在一个置换
    • 若不存在置换，则不可合一
    • 若存在，则这些置换的合成为其合一者
    • 若两个对应项都是变量时，置换选作用一个变量替代另一个变量，则这样得到的合一者为 mgu
  • 注意算法中可合一的表达式要用表结构来表示，如把 $P(x,f(A,y))$ 表示为 $(P\ x\ f\ A\ y)$
  • 可合一时返回 mgu，否则返回 FAIL
谓词逻辑归结过程
  • 判断是否可归结以及确定归结式：
  • 对于子句 $C1'\or L1$ 和 $C2'\or L2$ ，若 $L1$ 与 $\sim L2$ 可合一，且 $s$ 是其合一者，则 $(C1'\or C2')s$ 是其归结式

设 $C1$ 和 $C2$ 为两个不同的子句，子句中的变量已标准化，采用文字集的形式来表示子句（即文字之间理解为析取关系），则有： $C1={C1i}(i=1,2,...,n)$

$C2={C2j}(i=1,2,...,m)$

再设 ${L1k}$ 和 ${L2k}$ 分别为 $C1$ 和 $C2$ 的两个子集

若 ${L1k}$ 和 ${\sim L2k}$ 的并集存在一个 mgu s，则两个子句的归结式为

$$ C={ { C1i } - { L1k } } s \cup { { C2j } -{ L2k} } s $$

也可看出归结式不唯一
