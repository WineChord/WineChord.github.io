Notes for CS 276 11 Probabilistic Information Retrieval

发表于 2019-02-21 | 分类于 CS 276 IR |

Probabilistic Information Retrieval

  • From boolean to ranked retrieval ... in two steps

  • Ranked retrieval

  • Why probabilities in IR?

    • Probabilities provide a principled foundation for uncertain reasoning

  • Probabilistic IR topics

    1. Classical probabilistic retrieval model
    2. Bayesian networks for text retrieval
    3. Language model approach to IR
    • Probabilistic methods are one of the oldest but also one of the currently hot topics in IR

  • The document ranking problem

    • Ranking method is the core of modern IR systems:
    • Idea: Rank by probability of relevance of the document w.r.t. information need
      • $P(R=1|document_i,query)$

  • Bayes' Rule

    • $P(A|B)=\frac{p(B|A)p(A)}{p(B)}=\frac{p(B|A)p(A)}{\sum_{X=A,\bar{A}}p(B|X)p(X)}$

  • Odds: $O(A)=\frac{p(A)}{p(\bar{A})}=\frac{p(A)}{1-p(A)}$

  • Probability Ranking Principle

    • Let $x$ represent a document in the collection
    • Let $R$ represent relevance of a document w.r.t. given (fixed)
    • query and let R=1 represent relevant and R=0 not relevant
    • Need to find $p(R=1|x)$ -- probability that a document $x$ is relevant
    • $p(R=1|x)=\frac{p(x|R=1)p(R=1)}{p(x)}$
    • $p(R=0|x)=\frac{p(x|R=0)p(R=0)}{p(x)}$
    • $p(R=0|x)+p(R=1|x)=1$
    • Simple case: no selection costs or other utility concerns that would differentially weight errors
    • PRP in action: Rank all documents by $p(R=1|x)$
    • Theorem: Using the PRP is optimal, in that it minimizes the loss(Bayes risk) under 1/0 loss
      • Provable if all probabilities correct
    • More complex case: retrieval costs
      • Let $d$ be a document
      • $C$ -- cost of not retrieving a relevant document
      • $C'$ -- cost of retrieving a non-relevant document
    • Probability Ranking Principle: if

    $C'\cdot p(R=0|d)-C\cdot p(R=1|d)\le C'\cdot p(R=0|d')-C\cdot p(R=1|d')$

    • for all d' not yet retrieved, then $d$ is the next document to be retrieved

    • won't further consider cost/utility from now on

    • How to compute all those probabilities?

      • Do not know exact probabilities, have to use estimates
      • Binary Independence Model (BIM) -- which we discuss next -- is the simplest model

    • Questionable assumptions

      • "Relevance" of each document is independent of relevance of other documents
        • Really, it's bad to keep on returning duplicates
      • Boolean model of relevance
      • That one has a single step information need
        • Seeing a range of results might let user refine query

  • Probabilistic Retrieval Strategy

    • Estimate how terms contribute to relevance
      • How do other things like term frequency and document length influence your judgments about document relevance?
    • Combine to find document relevance probability
    • Order documents by decreasing probability

  • Probabilistic Ranking

    • Basic concept:

      For a given query, if we know some documents that are relevant, terms that occur in those documents should be given greater weighting in searching for other relevant documents.

      By making assumptions about the distribution of terms and applying Bayes Theorem, it is possible to derive weights theoretically.

  • Binary Independence Model

    • Traditionally used in conjunction with PRP

    • "Binary" = Boolean: documents are represented as binary incidence vectors of terms (cf. IIR Chapter 1):

      • $\vec{x}=(x_1,...,x_n)$
      • $x_i=1$ iff term $i$ is present in document $x$

    • "Independence": terms occur in documents independently

    • Different documents can be modeled as the same vector

    • Queries: binary term incidence vectors

    • Given query $q$,

      • for each document $d$ need to compute $p(R|q,d)$
      • replace with computing $p(R|q,x)$ where $x$ is binary term incidence vector representing $d$
      • Interested only in ranking

    • Will use odds and Bayes' Rule $$ O(R|q,\vec{x}) =\frac{p(R=1|q,\vec{x})}{p(R=0|q,\vec{x})}=\frac{\frac{p(R=1|q)p(\vec{x}|R=1,q)}{p(\vec{x}|q)}}{\frac{p(R=0|q)p(\vec{x}|R=0,q)}{p(\vec{x}|q)}}\\ = \frac{p(R=1|q)}{p(R=0|q)}\cdot \frac{p(\vec{x}|R=1,q)}{p(\vec{x}|R=0,q)} $$

      • Using Independence Assumption:

      $$ \frac{p(\vec{x}|R=1,q)}{p(\vec{x}|R=0,q)}=\prod_{i=1}^n\frac{p(x_i|R=1,q)}{p(x_i|R=0,q)} $$

      $$ O(R|q,\vec{x})=O(R|q)\cdot\prod_{i=1}^n\frac{p(x_i|R=1,q)}{p(x_i|R=0,q)} $$

      • Since $x_i$ is either 0 or 1:

      $$ O(R|q,\vec{x})=O(R|q)\cdot \prod_{x_i=1}\frac{p(x_i=1|R=1,q)}{p(x_i=1|R=0,q)}\cdot\prod_{x_i=0}\frac{p(x_i=0|R=1,q)}{p(x_i=0|R=0,q)} $$

      • Let $p_i=p(x_i=1|R=1,q)$; $r_i=p(x_i=1|R=0,q)$
      • Assume, for all terms not occurring in the query ($q_i=0$) $p_i=r_i$

      $$ O(R|q,\vec{x})=O(R|q)\cdot \prod_{x_i=1\\q_i=1}\frac{p_i}{r_i}\cdot \prod_{x_i=0\\q_i=1}\frac{(1-p_i)}{(1-r_i)} $$

      (All matching terms $\cdot$ Non-matching query terms) $$ \begin{align*} O(R|q,\vec{x})&=O(R|q)\cdot \prod_{x_i=1\\q_i=1}\frac{p_i}{r_i}\cdot \prod_{x_i=1\\q_i=1}(\frac{1-r_i}{1-p_i}\cdot \frac{1-p_i}{1-r_i})\prod_{x_i=0\\q_i=1}\frac{1-p_i}{1-r_i}\\ &=O(R|q)\cdot\prod_{x_i=q_i=1}\frac{p_i(1-r_i)}{r_i(1-p_i)}\cdot \prod_{q_i=1}\frac{1-p_i}{1-r_i} \end{align*} $$ (All matching terms $\cdot​$ All query terms)

      $\prod_{x_i=q_i=1}\frac{p_i(1-r_i)}{r_i(1-p_i)}​$ is the only quantity to be estimated for rankings

      Retrieval Status Value: $$ RSV=log\prod_{x_i=q_i=1}\frac{p_i(1-r_i)}{r_i(1-p_i)}=\sum_{x_i=q_i=1}log\frac{pi(1-r_i)}{r_i(1-p_i)} $$ All boils down to computing RSV. $$ RSV=\sum_{x_i=q_i=1}c_i;\ c_i=log\frac{p_i(1-r_i)}{r_i(1-p_i)} $$ The $c_i$ are log odds ratios

      They function as the term weights in this model

      $\rightarrow$ So, how do we compute $c_i$'s from our data?

    • Estimating RSV coefficients in theory

    • For each term $i$ look at this table of document counts:

      Documents Relevant Non-Relevant Total
      $x_i=1$ $s$ $n-s$ $n$
      $x_i=0$ $S-s$ $N-n-S+s$ $N-n$
      Total $S$ $N-S$ $N$

    • Estimates:

    $$ p_i\approx \frac{s}{S}\ \ \ r_i\approx\frac{(n-s)}{(N-S)} $$

    $$ c_i\approx K(M,n,S,s)=log\frac{s/(S-s)}{(n-s)/(N-n-S+s)} $$

    For now, assume no zero terms. Remember smoothing.

  • Estimation -- key challenge

    • If non-relevant documents are approximated by the whole collection, then $r_i$ (prob. of occurrence is non-relevant documents for query) is $n/N$ and

    $$ log\frac{1-r_i}{r_i}=log\frac{N-n-S+s}{n-s}\approx log\frac{N-n}{n}\approx\frac{N}{n}=IDF! $$

  • Collection vs. Document frequency

    • Collection frequency of $t$ is the total number of occurrences of $t$ in the collection (incl. multiples)
    • Document frequency is number of docs $t$ is in

  • Estimation -- key challenge

    • $p_i$ (probability of occurrence in relevant documents) cannot be approximated as easily

    • $p_i$ can be estimated in various ways:

      • from relevant documents if you know some
        • Relevance weighting can be used in a feedback loop
      • constant (Croft and Harper combination match) -- then just get idf weighting of terms (with $p_i$=0.5)

      $$ RSV=\sum_{x_i=q_i=1}log\frac{N}{n_i} $$

      • proportional to prob. of occurrence in collection

  • Probabilistic Relevance Feedback

    1. Guess a preliminary probabilistic description of $R=1$ documents; use it to retrieve a set of documents

    2. Interact with the user to refine the description:

      learn some definite members with $R=1$ and $R=0$

    3. Re-estimate $p_i$ and $r_i$ on the basis of these

      • If $i$ appears in $V_i$ within set of documents $V$: $p_i=|V_i|/|V|$

      • Or can combine new information with original guess (use Bayesian prior): $$ p_i^{(2)}=\frac{|V_i|+\kappa p_i^{(1)}}{|V|+\kappa} $$ $\kappa$ is prior weight

    4. Repeat, thus generating a succession of approximations to relevant documents

  • Iteratively estimating $p_i$ and $r_i$ (=Pseudo-relevance feedback)

    1. Assume that $p_i$ is constant over all $x_i$ in query and $r_i$ as before

      • $p_i=0.5$ (even odds) for any given doc

    2. Determine guess of relevant document set:

      • $V$ is fixed size set of highest ranked documents on this model

    3. We need to improve our guesses for $p_i$ and $r_i$, so

      • Use distribution of $x_i$ in docs in $V$. Let $V_i$ be set of documents containing $x_i$
        • $p_i=|V_i|/|V|$
      • Assume if not retrieved then not relevant
        • $r_i=(n_i-|V_i|)/(N-|V|)$

    4. Go to 2. until converges then return ranking

  • PRP and BIM

    • Getting reasonable approximations of probabilities is possible
    • Requires restrictive assumptions:
      • Term independence
      • Terms not in query don't affect the outcome
      • Boolean representation of documents/queries/relevance
      • Document relevance values are independent
    • Some of these assumptions can be removed
    • Problem: either require partial relevance information or seemingly only can derive somewhat inferior term weights

  • Removing term independence

    • In general, index terms aren't independent
    • Dependencies can be complex

  • Second step: Term frequency

    • Right in the first lecture, we said that a page should rank higher if it mentions a word more
      • Perhaps modulated by things like page length
    • We might want a model with term frequency in it
    • We'll see a probabilistic one next time -- BM25
    • Quick summary of vector space model

Notes for CS 276 6 Scoring, Term Weighting and the Vector Space Model

发表于 2019-02-21 | 分类于 CS 276 IR |

主要讲给文档评等级，和之前的 "Boolean queries" 相区别。

重点有：Vector space model 和 tf-idf

  • Ranked retrieval

    • 之前讲的都是二元查询：一个文档有或者没有，但是这不适用于大众化的查询

  • 二元查询出现的问题

    • 结果时好时坏：正确匹配时结果很多，但是一旦不匹配一个结果都没有

  • 引出评级检索模型

    • 查询语句更加随意（Free text queries），给出的是一系列有序的回答

  • 只给出最好的 $k$ 个结果

  • 打分数是评级检索中的基础

    • 这些分数代表了文档和查询之间的符合程度

  • 需要一种手段来给一对（查询/文档）来打分

    • 查询中只有一个词？那么如果文档中没有这个词，评分为0
    • 文档中该词出现的次数越多，评分越高

  • 方法1：Jaccard coefficient

    • $jaccard(A,B)=\frac{|A\cap B|}{|A\cup B|}$
    • $jaccard(A,A)=1$
    • $jaccard(A,B)=0\ if\ A\cap B=0$
    • 该方法的问题：
      • 没有考虑到关键词的频率（term frequency）
      • 一些不常见的关键词提供了更多的信息，这个方法没有考虑这一点
      • 需要更加成熟的方式来归一化长度
      • 之后会用 $\frac{|A\cap B|}{\sqrt{|A\cup B|}}$

  • 关键词-文档矩阵

    • 每个文档用一个向量来表示，这个向量中每个值分别代表了一种关键字出现的次数

  • 词袋模型（Bag of words model）

    • 向量表示没有考虑关键字在文档中的次序
    • 这称为词袋模型

  • 关键词频率 tf（term frequency）

    • $tf_{t,d}$ 表示关键词 $t$ 在文档 $d$ 中出现的频率
    • 如何用这个定义来计算分数？
    • 相关性并不正比于词频

  • 对数频率权重（Log-frequency weighting）

$$ w_{t,d}= \begin{cases} 1+log_{10}tf_{t,d},\ &if\ tf_{t,d}\gt0\ 0,&otherwise \end{cases} $$

$$ score=\sum_{t\in q\cap d}(1+log\ tf_{t,d}) $$

如果没有一个查询的关键词出现在文档中，分数为0

  • 文档频率（Document frequency）

    • 不常见的关键字提供更多信息
    • 考虑在查询语句中不常见的关键词
    • 假如一个文档包含了这个不常见的关键词，那么这个文档很可能与查询相关
    • 因此我们希望提高不常见的关键词的权重
    • 而对于频率较高的关键词，也应有较高的权重，但是要低于不常见的词的权重
    • 用文档频率来进行表征（document frequency）

  • idf weight

    • $df_t$ 是关键词 $t$ 的文档频率：即多少个文档包含关键词 $t$
      • 是对 $t$ 信息多少的反向评估
      • $df_t\le N$ （$N$ 是文档的总数）
    • 定义 idf (inverse document frequency) 反向文档频率如下：
      • $idf_t=log_{10}(\frac{N}{df_t})$
      • 使用对数是为了减弱 $idf$ 的影响

  • idf 对排名的影响

    • idf 对单关键字的查询没有影响
    • 影响多关键字查询，比如查询 "capricious person"，idf 会使 "capricious" 出现的次数对评分影响重一些

  • Collection vs. Document frequency

    • collection frequency 指的是一个关键字在整个文集中出现的次数
    • df 指的是该关键字出现在了多少篇文章当中

  • tf-idf weighting

    • $w_{t,d}=log(1+tf_{t,d})\times log_{10}(\frac{N}{df_t})$
    • 是 $tf$ 和 $idf$ 的乘积
    • 是一个很有名的权重方案
    • 当文档中词频上升或关键词的稀有程度较高时，该值较大

  • Score for a document given a query

    • $Score(q,d)=\sum_{t\in q\cap d}tf.idf_{t,d}$
    • 变种有很多：比如 $tf$ 是怎么计算的啊（是否取对数），查询中的关键字是否也参与权重计算当中啊...

  • 现在每一个文档都被表示成了使之向量，向量中的每一个值是 $tf-idf$ 权重 $\in \mathbb{R}^{|V|}$

  • 文档就变成了向量！

    • 至此我们有了 $|V|$ 维向量空间
    • 关键字是空间的坐标轴
    • 文档是这个空间中的向量
    • 把这个应用到网络搜索引擎中去时，可以达到非常高的维度
    • 同时也是非常稀疏的向量：很多值都是0

  • 查询语句也可以作为向量

    • 关键思想1：将查询语句也表示成空间中的向量
    • 关键思想2：根据文档与查询在空间中的临近程度来对文档进行排序
    • proximity = similarity of vectors
    • proximity $\approx$ inverse of distance
    • 即文档和查询语句之间的相似程度

  • 形式化向量空间的临近度

    • 第一想法：两点之间的距离？（两个向量末端点的距离）
    • 欧几里得距离？？
    • 并不好：对于不同长度的向量而言，这个距离过大了

  • 使用角度来衡量！

    • 想象一下：把一个文档复制一下加到它自己后面，复制前后文档的相似度应该是一样的！
    • 关键思路：根据文档和查询之间的夹角大小来对文档进行排序

  • 从角度到余弦值

    • 该角度距离可以用余弦值的大小来衡量！
    • 因为余弦函数在 $[0^{\circ}, 180^{\circ}]$ 内单调递减！

  • 长度归一化（Length normalization）

    • 使用 $L_2$ 范数：$||\vec{x}||_2=\sqrt{\sum_i{x_i^2}}$

  • cosine(query, document)

$$ cos(\vec{q},\vec{d})=\frac{\vec{q}\cdot \vec{d}}{|\vec{q}||\vec{d}|}=\frac{\sum_{i=1}^{|V|}q_id_i}{\sqrt{\sum_{i=1}^{|V|}q_i^2}\sqrt{\sum_{i=1}^{|V|}d_i^2}} $$

  • $q_i$ 是第i个关键字在查询中的 tf-idf 权重
  • $d_i$ 是第i个关键字在文档中的 tf-idf 权重
  • idf 中 log 的底是多少是无所谓的

总结：向量空间排名

  • 把查询语句表示成一个 tf-idf 向量
  • 把每个文档表示成一个 tf-idf 向量
  • 计算查询语句与每一个文档的余弦相似度
  • 根据余弦相似度来对文档进行排名
  • 给用户分数最高的 $k$ （比如 $k=10$）个答案

参考资料：starter code , class results

两种基本的压缩方法：

  1. 传统方法：bit-level such as Huffman, Golumb, Gamma, Delta, Rice ...
  2. 对齐方法："Byte aligned" or "Word aligned" encodings, where roughly there are stretches of bits that you can use directly as numbers, and you mainly access them by masking and shifting

后一种打败了前一种方法，虽然不是压缩率最高的，但是后一种方法的解压速度更快，并且更容易实现

在索引压缩中，考虑两个点：

  1. size
  2. speed

在某种程度上，减小大小可以提高速度，但是不能太过，字节对齐所带来的速度提升要比它增加的大小要划算得多。

这个 lab 主要关注于大小

对于任何一个压缩方法，在中间都必须有一组字节，比如放在 ByteArrayOutputStream 或者 byte[] 中

Notes for CS 276 5 Index Compression

发表于 2019-02-21 | 分类于 CS 276 IR |

主要讲索引压缩，先讲了一下数据集的规模信息，一些理论，然后分别讲了 "Dictionary compression" 和 "Postings compression"

  • Collection statistics in more detail (with RCV1)
  • Dictionary compression
  • Postings compression
  • Why compression?
    • save memory
    • increase speed
    • increase speed of data transfer from disk to memory
  • Why compression for inverted indexes?
    • Dictionary
      • small enough to keep in main memory
      • so small that can keep some postings lists in memory too
    • Postings files
      • reduce disk space needed
      • decrease time needed to read postings lists from disk
      • large se keep a significant part of the postings in memory
  • Lossless vs. lossy compression
    • lossless: mostly we do in IR
    • lossy: case folding/stop words/stemming/number elimination
  • Vocabulary vs. collection size
    • How big is the term vocabulary?
    • can assume an upper bound?
    • In practice, the vocabulary will keep growing with the collection size
    • Heaps' law: $M=kT^b$
    • $M$ is the size of the vocabulary, $T$ is the number of tokens in the collection
    • Typical values: $30\le k\le 100$ and $b\approx 0.5$
    • In a log-log plot of vocabulary size M vs. T, Heaps' law predicts a line with slope about $\frac{1}{2}$
      • An empirical finding
  • Zipf's law
    • Heaps' law gives the vocabulary size in collections
    • also study the relative frequencies of terms
    • In natural language, there are a few very frequent terms and very many very rare terms
    • Zipf's law: The ith most frequent term has frequency proportional to 1/i
    • $cf_i\propto \frac{1}{i} = \frac{K}{i}$ where $K$ is a normalizing constant
    • $cf_i$ is collection frequency: the number of occurrences of the term $t_i$ in the collection
  • Zipf consequences
    • If the most frequent term (the) occurs $cf_1$ times
      • then the second most frequent term (of) occurs $cf_1/2$ times
      • the third most frequent term (and) occurs $cf_1/3$ times
    • Equivalent: $cf_i=K/i$ where $K$ is a normalizing factor, so
      • $log(cf_i)=log(K)-log(i)$
      • Linear relationship between $log(cf_i)$ and $log(i)$
    • Another power law relationship
  • Compression
    • Basic Boolean index only
    • No study of positional indexes
    • consider compression schemes

Dictionary Compression

  • Why compress the dictionary?
    • Search begins with the dictionary
    • want to keep it in memory
    • Memory foorprint competition with other applications
  • Dictionary storage -- first cut
    • Array of fixed-width entries
      • ~400,000 terms; 28 bytes/term = 11.2 MB
  • Fixed-width terms are wasteful
  • Compressing the term list: Dictionary-as-a-String
    • Store dictionary as a (long) string of characters:
      • Pointer to next word shows end of current word
      • Hope to save up to 60% of dictionary space
  • Space for dictionary as a string
    • 4 bytes per term for Freq.
    • 4 bytes per term for pointer to Postings
    • 3 bytes per term pointer
    • Avg. 8 bytes per term in term string
    • 400K terms $\times$ 19 $\Rightarrow$ 7.6 MB (against 11.2 MB for fixed width)
  • Blocking
    • Store pointers to every kth term string
    • Need to store term lengths (1 extra byte)
  • Dictionary search without blocking
  • Dictionary search with blocking
  • Front coding
    • Sorted words commonly have long common prefix -- store differences only
    • $\color{red}{8}$$automata$$\color{red}{8}$$automate$$\color{red}{9}$$automatic$$\color{red}{10}$$automation$
    • $\rightarrow$ $\color{red}{8}$$automat^*a$$\color{red}{1}$$\Diamond e$$\color{red}{2}$$\Diamond ic$$\color{red}{3}$$\Diamond ion$
    • $^*$ encodes $automat$ $\color{red}{1}$ is the extra length beyond $automat$

Postings Compression

  • Postings compression
    • Postings file is much larger than the dictionary, factor of at least 10.
    • Key desideratum: store each posting compactly
    • A posting for our purposes is a docID
    • For Reuters (800,000 documents), we would use 32 bits per docID when using 4-byte integers
    • Or, use $log_2800,000\approx 20$ bits per docID
    • Our goal: use far fewer than 20 bits per docID
  • Postings: two conflicting forces
    • A term like arachnocentric occurs in maybe one doc out of a million -- we would like to store this posting using $log_21M$ ~ 20 bits
    • A term like the occurs in virtually every doc, so 20 bits/posting is too expensive
      • Prefer 0/1 bitmap vector in this case
  • Postings file entry
    • current: store the list of docs containing a term in increasing order of docID
      • computer: 33,47,154,159,202...
    • Consequence: it suffices to store gaps
      • 33,14,107,5,43
    • Hope: most gaps can be encoded/stored with far fewer than 20 bits
  • Variable length encoding
    • Aim:
      • For arachnocentric, we will use ~ 20 bits/gap entry
      • For the, we will use ~ 1 bit/gap entry
    • If the average gap for a term is G, we want to use ~$log_2G$ bits/gap entry
    • Key challenge: encode every integer (gap) with about as few bits as needed for that integer
    • This requires a variable length encoding
    • Variable length codes achieve this by using short codes for small numbers
  • Variable Bytes (VB) codes
    • For a gap value G, we want to use close to the fewest bytes needed to hold $log_2G$ bits
    • Begin with one byte to store G and dedicate 1 bit in it to be continuation bit c
    • if G $\le 127$, binary-encode it in the 7 available bits and set c=1
    • Else encode G's lower-order 7 bits and then use additional bytes to encode the higher order bits using the same algorithm
    • At the end set the continuation bit of the last byte to 1 (c=1) -- and for the other bytes c=0
  • Other variable unit codes
    • Instead of bytes, we can also use a different "unit of alignment": 32 bits (words), 16 bits, 4 bits (nibbles)
    • Variable byte alignment wastes space if you have many small gaps -- nibbles do better in such cases
    • Variable byte codes:
      • Used by many commerical/research systems
      • Good low-tech blend of variable-length coding and sensitivity to computer memory alignment matches (vs. bit-level codes, which we look at next)
    • There is also recent work on word-aligned codes that pack a variable number of gaps into one word
  • Unary code
    • Represent n as n 1s with a final 0.
    • e.g., unary code for 3 is 1110
  • Gamma codes
    • can compress better with bit-level codes
      • The Gamma code is the best known of these
    • Represent a gap G as a pair length and offset
    • offset is G in binary, with the leading bit cut off
      • e.g., 13 $\rightarrow$ 1101 $\rightarrow$ 101
    • length is the length of offset
      • e.g., 13 (offset 101), this is 3
    • We encode length with unary code: 1110
    • Gamma code of 13 is the concatenation of length and offset: 1110101
  • Gamma code properties
    • G is encoded using $2\lfloor logG\rfloor+1$ bits
      • Length of offset is $\lfloor logG\rfloor$ bits
      • Length of length is $\lfloor logG\rfloor +1$ bits
    • All gamma codes have an odd number of bits
    • Almost within a factor of 2 of best possible, $log_2G$
    • Gamma code is uniquely prefix-decodable, like VB
    • Gamma code can be used for any distribution
    • Gamma code is parameter-free
  • Gamma seldom used in practice
    • Machines have word boundaries -- 8,16,32,64 bits
      • Operations that cross word boundaries are slower
    • Compressing and manipulating at the granularity of bits can be slow
    • Variable byte encoding is aligned and thus potentially more efficient
    • Regardless of efficiency, variable byte is conceptually simpler at little additional space cost
  • Index compression summary
    • We can now create an index for highly efficient Boolean retrieval that is very space efficient
    • Only 4% of the total size of the collection
    • Only 10-15% of the total size of the text in the collection
    • However, we've ignored positional information
    • Hence, space savings are less for indexes used in preactice
      • But techniques substantially the same

Dictionary data structures

  • The dictionary DS stores the term vocabulary, doc frequency, pointers to each postings list ... in what data structure?
  • A naive dictionary
    • An array of struct: term/doc frequency/pointer to postings list
    • store efficiently? quickly look up?
  • Dict DS
    • Hashtables/Trees
  • Hashtables
    • Each vocabulary term is hashed to an integer
    • Pros:
      • Lookup is faster than tree: $O(1)$
    • Cons:
      • No easy way to find minor variants
      • No prefix search
      • if keep growing, need to rehash everything -- expensive
  • Tree: binary tree/B-tree
    • require a standard ordering of characters and hence strings ... but we typically have one
    • Pros:
      • Solves the prefix problem
    • Cons:
      • Slower: $O(logM)$ (requires balanced tree)
      • Rebalancing binary trees is expensive
        • But B-trees mitigate the rebalancing problem

Wild-Card Queries

  • mon*: find all docs containing any word beginning with "mon"
  • Easy with binary tree (or B-tree) lexicon: retrieve all words in range: mon $\le$ w $\lt$ moo
  • *mon: find words ending in "mon": harder
    • Maintain an additional B-tree for terms backwards
    • Can retrieve all words in range: nom $\le$ w $\lt$ non
  • Exercise: from this, how can we enumerate all terms meeting the wild-card query pro*cent?
  • Got an enumeration of all terms in the dictionary that match the wild-card query
  • Still have to look up the postings for each enumerated term
  • E.g., consider the query:
    • se*ate AND fil*er
    • may result in the execution of many Boolean AND queries
  • B-trees handle *'s at the end of a query term
    • handle co*tion ?
    • solu: transform wild-card queries so that the *'s occur at the end
    • This gives rise to the Permuterm Index
  • Permuterm index
    • For term hello, index under:
      • hello$,ello$h,llo$he,lo$hel,o$hell,$hello
      • where $ is a special symbol
    • Queries:
      • X lookup on X$ X* lookup on $X*
      • *X lookup on X$* *X* lookup on X*
      • X*Y lookup on Y$X* (Query=hel*o, X=hel,Y=o, lookup o$hel*)
      • X*Y*Z ???
  • Permuterm query processing
    • Rotate query wild-card to the right
    • Now use B-tree lookup as before
    • Permuterm problem: $\approx$ quadruples lexicon size
  • Bigram (k-gram) indexes
    • Enumerate all k-grams (sequence of k chars) occurring in any term
    • maintain a second inverted index from bigrams to dict terms that match each bigram
  • Processing wild-cards
    • Query mon* can now be run as
      • $m AND mo AND on
    • Gets terms that match AND version of our wildcard query
    • But we'd enumerate moon
    • Must post-filter these terms against query
    • Surviving enumerated terms are then looked up in the term-document inverted index
    • Fast, space efficient (compared to permuterm)
  • Processing wild-card queries
    • execute a Boolean query for each enumerated, filtered term
    • Wild-cards can result in expensive query execution (very large disjunctions)

Spelling Correction

  • Spell correction

    • Two principal uses
      • Correcting docs being indexed
      • Correcting user queries to retrieve "right" answers
    • Two main flavors
      • Isolated word
        • Check each word on its own for misspelling
      • Context-sensitive
        • Look at surrounding words

  • Document correction

    • needed for OCR'ed docs
      • rn $\rightarrow$ m
      • Can use domain-specific knowledge
    • But also: web pages and even printed material have typos
    • Goal: the dictionary contains fewer misspellings
    • But often we don't change the docs and instead fix the query-document mapping

  • Query mis-spellings

    • can either: Retrieve docs indexed by the correct spelling, OR
    • Return several suggested alternative queries with the correct spelling: Do you mean ... ?

  • Isolated word correction

    • Fundamental premise -- there is a lexicon from which the correct spellings come
    • Two basic choices for this
      • A standard lexicon
      • The lexicon of the indexed corpus
    • Given a lexicon and a character sequence Q, return the words in the lexicon closest to Q
    • What's "closest"?
      • Edit distance (Levenshtein distance)
      • Weighted edit distance
      • n-gram overlap

  • Edit distance

    • Given two strings $S_1$ and $S_2$, the minimum number of operations to convert one to the other
    • Operations are typically character-level
      • Insert, Delete, Replace, (Transposition 移项)
    • Generally found by dynamic programming

  • Weighted edit distance

    • weight of an operation depends on the character(s) involved
      • Meant to capture OCR or keyboard errors
      • e.g., m more likely to be mis-typed as n than as q
      • Therefore, replacing m by n is a smaller edit distance than by q
      • This may be formulated as a probability model
    • Requires weight matrix as input
    • Modify dp to handle weights

  • Using edit distances

    • Given query, first enumerate all character sequences within a preset(weighted) edit distance (e.g., 2)
    • Intersect this set with list of "correct" words
    • Show terms you found to user as suggestions
    • Alternatively
      • can lookup all possible corrections in out inverted index and return all docs ... slow
      • can run with a single most likely correction
    • The alternatives disempower the user, but save a round of interaction with the user

  • Edit distance to all dictionary terms?

    • Given a (mis-spelled) query -- do we compute its edit distance to every dictionary term?
      • Expensive and slow
      • Alternatives?
    • How to cut the set of candidate dictionary terms?
    • One possibility: use n-gram overlap
    • can also be used by itself for spelling correction

  • n-gram overlap

    • Enumerate all the n-grams in the query string as well as in the lexicon
    • Use the n-gram index (recall wild-card search) to retrieve all lexicon terms matching any of the query n-grams
    • Threshold by number of matching n-grams
      • Variants -- weight by keyboard layout, etc.

  • Example with trigrams

    • text: november
      • Trigrams: nov,ove,vem,emb,mbe,ber
    • query: december
      • Trigrams: dec,ece,cem,emb,mbe,ber
    • 3 trigrams overlap (of 6 in each term)
    • How to turn this into a normalized measure of overlap?

  • One option -- Jaccard coefficient

    • A commonly-used measure of overlap

    • $$ \frac{|X\cap Y|}{|X\cup Y|} $$

    • e.g., if J.C. $\gt$ 0.8, declare a match

  • Context-sensitive spell correction

    • Need surrounding context to catch this
    • First idea: retrieve dict terms close (in weighted edit distance) to each query term
    • Now try all possible resulting phrases with one word "fixed" at a time
    • Hit-based spelling correction: Suggest the alternative that has lots of hits

  • Another approach

    • Break phrase query into a conjunction of biwords
    • Look for biwords that need only one term corrected
    • Enumerate only phrases containing "common" biwords

  • General issues in spell correction

    • enumerate multiple alternatives for "Did you mean?"
    • need to figure out which to present to the user
      • The alternative hitting most docs
      • Query log analysis
    • more generally, rank alternatives probabilistically
      • $argmax_{corr}P(corr|query)$
      • From Bayes rule: it equals $argmax_{corr}P(query|corr)*P(corr)$

SOUNDEX

  • Class of heuristics to expand a query into phonetic equivalents

    • Language specific -- mainly for names
    • e.g., chebyshev $\rightarrow$ tchebycheff
    • Invented for the U.S. census ... in 1918

  • Typical algorithm

    • Turn every token to be indexed into a 4-character reduced form
    • Do the same with query terms
    • Build and search an index on the reduced forms
      • (when the query calls for a soundex match)

    1. Retain the first letter of the word

    2. Change all occurrences of the following letters to '0'(zero): "AEIOUHWY"

    3. Change letters to digits as follows:

      "B,F,P,V" $\rightarrow$ 1

      "C,G,J,K,Q,S,X,Z" $\rightarrow$ 2

      "D,T" $\rightarrow$ 3

      "L" $\rightarrow$ 4

      "M,N" $\rightarrow$ 5

      "R" $\rightarrow$ 6

    4. Remove all pairs of consecutive digits

    5. Remove all zeros from the resulting string

    6. Pad the resulting string with trailing zeros and return the first four positions, which will be of the form <uppercase letter><digit><digit><digit>

    • e.g., Herman becomes H655

  • Recall IIR 1 index construction

  • Key step: inverted file is sorted by terms

    • focus on this sort step: 100M items to sort

  • Scaling index construction

    • In-memory index construction does not scale
    • How to construct an index for very large collections?

  • Sort-based index construction

    • As we build the index, we parse docs one at a time
    • The final postings for any term are incomplete until the end
    • At 12 bytes per non-positional postings entry(term, doc, freq), demands a lot of space for large collections.
    • T = 100,000,000 in the case of RCV1
    • Thus: We need to store intermediate results on disk

  • Sort using disk as "memory"?

    • Too slow: many disk seeks
    • need external sorting algorithm

  • Bottleneck

    • Parse and build postings entries one doc at a time
    • Now sort postings entries by term (then by doc within each term)
    • Doing this with random disk seeks would be too slow

  • BSBI: Blocked Sorted-Based Indexing(Sorting with fewer disk seeks)

    • 12-byte (4+4+4) records (term, doc, freq)
    • These are generated as we parse docs
    • Must now sort 100M such 12-byte records by term
    • Define a Block ~10M such records
      • 10 such blocks
    • Basic idea of algorithm:
      • Accumulate postings for each block, sort, write to disk
      • Then merge the blocks into one long sorted order

  • Sorting 10 blocks of 10M records

    • First, read each block and sort within:
      • Qsort: 2NlnN expected steps
    • 10 times this estimate -- gives us 10 sorted runs of 10M records each
    • Done straightforwardly, need 2 copies of data on disk
      • can be optimized

  • How to merge the sorted runs?

    • Can do binary merges
    • more efficient to do a multi-way merge

  • Remaining problem with sort-based algorithm

    • assumption: can keep the dictionary in memory
    • dictionary to implement a term to term ID mapping
    • but: could work with term instead of termID
      • intermediate files become very large

  • SPIMI: Single-Pass In-Memory Indexing

    • Key idea1: Generate separate dictionaries for each block -- no need to maintain term-termID mapping across blocks
    • Key idea2: Don't sort. Accumulate postings in postings lists as they occur
    • With these two ideas we can generate a complete inverted index for each block
    • These separate indexes can then be merged into one big index

  • SPIMI: Compression

    • makes it even more efficient
      • Compression of terms
      • Compression of postings

  • Distributed indexing

    • web-scale indexing: use a distributed computing cluster

  • Web search engine data centers

  • Massive data centers

  • Distributed indexing

    • Maintain a master machine directing the indexing job
      • -- considered "safe"
    • Break up indexing into sets of (parallel) tasks
    • Master machine assigns each task to an idle machine from a pool

  • Parallel tasks

    • use two sets of parallel tasks
      • Parsers/Inverters
    • Break the input document collection into splits
    • Each split is a subset of documents(corresponding to blocks in BSBI/SPIMI)

  • Parsers

    • Master assigns a split to an idle parser machine
    • Parser reads a document at a time and emits (term, doc) pairs
    • Parser writes pairs into j partitions
    • Each partition is for a range of terms' first letters
      • a-f, g-p, q-z ( j=3 )
    • Now to complete the index inversion

  • Inverters

    • An inverter collects all (term, doc) pairs (= postings) for one term-partition
    • Sorts and writes to postings lists

  • MapReduce

    • Google indexing system consists of a number of phases, each implemented in MapReduce.
    • Index construction was just one phase
    • Another phase: transforming a term-partitioned index into a document-partitioned index
      • Term-partitioned: one machine handles a subrange of terms
      • Document-partitioned: one machine handles a subrange of documents
    • in the web part of the course, most search engines use a document-partitioned index

  • Schema for index construction in MapReduce

    • Schema of map and reduce functions
    • map: input $\rightarrow$ list(k,v), reduce: (k, list(v)) $\rightarrow$ output
    • Instantiation of the schema for index construction
    • map: collection $\rightarrow$ list(termID, docID)
    • reduce: (<termID1, list(docID)>, <termID2, list(docID)>,...) $\rightarrow$ (postings list1, postings list2,...)

  • Dynamic indexing

    • all above... are static
    • Doc being added or deleted or modified

  • Simplest approach

    • Maintain "big" main index
    • New docs go into "small" auxiliary index
    • Search across both, merge results
    • Deletions
      • Invalidation bit-vector for deleted docs
      • Filter docs output on a search result by this invalidation bit-vector
    • Periodically, re-index into one main index

  • Issues with main and auxiliary indexes

    • Problem of frequent merges -- you touch stuff a lot
    • Poor performance during merge
    • Assumption for the rest of the lecture: The index is one big file
    • In reality: Use a scheme somewhere in between(e.g., split very large postings lists, collect postings lists of length 1 in one file etc.)

  • Logarithmic merge

    • Maintain a series of indexes, each twice as large as the previous one
    • Keep smallest ($Z_0$) in memory
    • Larger ones ($I_0,I_1,...$) on disk
    • If $Z_0$ gets too big ($\gt n$), write to disk as $I_0$
    • or merge with $I_0$ (if $I_0$ already exists) as $Z_1$
    • Either write merge $Z_1$ to disk as $I_1$ (if no $I_1$)
    • Or merge with $I_1$ to form $Z_2$

  • Further issues with multiple indexes

    • Collection-wide statistics are hard to maintain

  • Dynamic indexing at search engines

    • All the large search engines now do dynamic indexing
    • Their indices have frequent incremental changes
    • But they also periodically reconstruct the index from scratch

  • Other sorts of indexes

    • Positional indexes
      • Same sort of sorting problem ... just larger
    • Building character n-gram indexes:
      • As text is parsed, enumerate n-grams
      • For each n-gram, need pointers to all dictionary terms containing it -- the "postings"
      • Note that the same "postings entry" will arise repeatedly in parsing the docs -- need efficient hashing to keep track of this

好，现在问题变成了：同时索引两个关键字，这两个关键字之间相距小于 $k$

实现思路：在上一步的基础上，先找到共同的文档，然后在文档中找复合条件的距离。

实现的关键是怎么在文档中找符合条件的距离？

然而这种思路是错误的，会漏掉一些 pp2 在 pp1 前的情况，参考 IIR Figure 2.12 后，提取其算法思路如下：

  1. 首先固定 pp1，将所有距离小于 $k$ 的 pp2 储存在一个列表 $l$ 中，直到 pp2 与 pp1 的距离超过了 $k$。

  2. 此时剔除 $l$ 前面与 pp1 不符合距离小于等于 $k$ 的元素，然后将剩下的添加到答案中。

  3. 更新 pp1 为下一个元素

Introducing Information Retrieval and Web Search

  • Def.
    • IR is finding material(usually documents) of an unstructured nature(usually text) that satisfies an information need from within large collections(usually stored on computers).
    • Collection: A set of documents
    • Goal: Retrieve documents with info. that is relevant to the user's info. need and helps the user complete a task.
  • Eval: How good are the retrieved docs?
    • Precision: Fraction of retrieved docs that are relevant to the user's info. need
    • Recall: Fraction of relevant docs in collection that are retrieved.

Term-document incidence matrices

  • using the number of appearances in each document to represent a term.

The Inverted Index

  • For each term t, store a list of all documents that contains t.

    • Identify each doc by a docID

  • Need variable-size postings lists (inside it is sorted by docID)

    • On disk: a continuous run of postings.
    • In memory: linked lists or variable length arrays

  • Inverted index construction

    • Documents to be indexed
    • $\underrightarrow{Tokenizer}$
    • Token stream
    • $\underrightarrow{Linguistic modules}$
    • Modified tokens
    • $\underrightarrow{Indexer}$
    • Inverted index

  • Initial stages of text processing

    • Tokenization
    • Normalization
    • Stemming
    • Stop words

  • Indexer steps

    • Token sequence
    • Sort (by terms and then docID)
    • Dictionary & Postings (Doc. frequency info is added)

  • Problems taken into consideration

    • efficient index
    • storage

Query processing with an inverted index

  • Query processing: AND
    • "Merge" the two postings: O(x+y)
    • Crucial: postings sorted by docID

The Boolean Retrieval Model & Extended Boolean Models

  • Boolean queries: Exact match
    • queries using AND, OR and NOT to join query terms
      • View each doc as a set of words
      • Is precise
  • Query optimization
    • Process in order of increasing freq
      • start with smallest set, then keep cutting further

Phrase queries and positional indexes

  • to answer queries such as "Stanford university"
  • It is no longer suffices to store only <term: docs> entries
  • A first attempt: Biword indexes
    • Index every consecutive pair of terms in the text as a phrase
    • Problems:
      • False positives
      • Index blowup due to bigger dictionary
  • Solution 2: Positional indexes

<term, number of docs containing term;

doc1: position1, position2...;

doc2: position1, position2...;

etc.>

  • Processing a phrase query
    • Extract inverted index entries for each distinct term
    • Merge their doc
  • Proximity queries
  • Positional index size
    • Although expands postings storage substantially, it is now standardly used because of the power and usefulness of phrase and proximity queries.
  • Rules of thumb
    • 2-4 as large as a non-positional index
    • size 35-50% of volume of original text
  • Combination schemes
    • can be profitably combined

Structured vs. Unstructured Data

  • Structured: info. in tables
  • Unstructured: free text
  • Semi-structured: has distinctly identified zones

多媒体技术的定义，特征，关键技术

定义：

  • 计算机综合处理多种媒体信息：文本，图形，图像，音频和视频，使多种信息建立逻辑连接，集成为一个系统并具有交互性。
    • 综合处理声，文，图信息
    • 集成性，交互性
    • 总之：信息载体多样性，集成性，交互性

关键技术：

  • 视频音频信号获取技术
    • 多媒体计算机和传统计算机相比就是能处理声文图，所以首先要解决视音频信号获取技术
  • 多媒体数据压缩编码和解码技术
    • 压缩编解码解决的是文件量过大的问题
  • 视频音频数据的实时处理和特技
  • 视频音频数据的输出技术
    • 视频输出：多个视频窗口，画中画电视
    • 音频输出：单声道$\rightarrow$ 环绕立体声

即：视音频的信号获取/实时处理和特技/输出/压缩编解码

DPCM编码原理

  • 差分量化编码是利用样本与样本之间存在的信息冗余度来进行编码的一种数据压缩技术。其根据过去的样本来估计下一个样本信号的幅度大小，此估计值为预测值，然后对实际信号值与预测值之间的差进行量化编码，从而减少了表示每个样本信号的位数
  • 即存储或传送的是差值

参数编码的国际标准

音频编码的分类：

  • 基于音频数据的统计特性进行编码：波形编码
  • 基于音频的声学参数：参数编码
  • 基于人的听觉特性进行编码

  • 预测方式：前向/后向，短时/长时
